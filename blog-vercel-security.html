      Vercel deploy — bezbednosna podesavanja za vibe kodere

      Kompletni vodic: vercel.json headeri sa pravim vrednostima, env varijable po okruzenjima, preview deploy zastita, Edge/Serverless bezbednost, domen/SSL i deploy checklista

      1. Zasto Vercel zahteva dodatnu konfiguraciju

      Vercel je najpopularnija platforma za deploy Next.js, React i statik sajtova. AI alati kao Cursor, Claude Code i v0 generisu kod koji se deploy-uje na Vercel jednim klikom. Problem je sto taj jedan klik ne ukljucuje bezbednosnu konfiguraciju.

      Evo sta Vercel automatski obezbedjuje: SSL/TLS sertifikat (Let's Encrypt), globalni CDN sa edge lokacijama, DDoS zastitu na mreznom nivou, automatske build-ove iz Git-a i izolaciju izmedju projekata.

      A evo sta Vercel ne radi za vas: ne dodaje nijedan bezbednosni HTTP header, ne enkriptuje env varijable po defaultu, ne ogranicava pristup preview deploy-ovima i ne implementira rate limiting na vasim API rutama.

      Kljucna razlika: Vercel vam daje bezbednu infrastrukturu. Ali konfiguracija vase aplikacije — headeri, tajne, pristup — je potpuno vasa odgovornost. Ovaj vodic pokriva svaku od tih stavki sa tacnim konfiguracijama koje mozete kopirati.

      2. Bezbednosni headeri u vercel.json — kompletna konfiguracija

      Ovo je najvazniji deo. Vercel ne dodaje nijedan security header po defaultu. Bez vercel.json fajla, vas sajt nema CSP, nema HSTS, nema clickjacking zastitu — nista. Ovo je produkcijski vercel.json sa svim potrebnim headerima:

      { "headers": [ { "source": "/(.*)", // Primenjuje se na SVE rute "headers": [ { "key": "Strict-Transport-Security", "value": "max-age=31536000; includeSubDomains; preload" }, { "key": "Content-Security-Policy", "value": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self'; frame-ancestors 'none'" }, { "key": "X-Frame-Options", "value": "DENY" }, { "key": "X-Content-Type-Options", "value": "nosniff" }, { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }, { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" }, { "key": "Cross-Origin-Opener-Policy", "value": "same-origin" } ] } ] }

      Svaki header ima specificnu ulogu:

      • Strict-Transport-Security — kaze browseru da vas sajt koristi iskljucivo HTTPS narednih 365 dana (max-age=31536000). includeSubDomains stiti i subdomene. preload omogucava upis u hromovu ugradjenu listu.
      • Content-Security-Policy — najslozeniji i najvazniji header. default-src 'self' znaci da se svi resursi ucitavaju samo sa vaseg domena. Svaka direktiva (script-src, style-src, font-src) precizira izuzetke. frame-ancestors 'none' zamenjuje X-Frame-Options u modernim browserima.
      • X-Frame-Options: DENY — sprecava da bilo koji sajt ucita vas sajt u iframe. Stiti od clickjacking napada gde napadac postavi nevidljivi iframe preko dugmeta.
      • X-Content-Type-Options: nosniff — sprecava browser da “pogadja” MIME tip. Bez ovoga, browser moze da interpretira tekstualni fajl kao JavaScript.
      • Referrer-Policy — strict-origin-when-cross-origin salje pun URL samo za isti origin; za spoljne linkove salje samo domen. Stiti od curenja URL parametara (tokeni, session ID-jevi).
      • Permissions-Policy — eksplicitno onemogucava pristup kameri, mikrofonu i geolokaciji. Cak i ako napadac ubaci skriptu, ne moze aktivirati ove senzore.
      • Cross-Origin-Opener-Policy — izoluje vas tab od drugih tabova. Sprecava Spectre-tipa napade gde drugi tab pristupa memoriji vaseg taba.

      CSP za sajt sa Google Ads

      Ako koristite Google AdSense, morate prosiriti CSP da dozvoli Google domene. Ovo je cesta situacija za vibe kodere koji monetizuju projekte:

      "value": "default-src 'self'; script-src 'self' 'unsafe-inline' https://*.googlesyndication.com https://*.g.doubleclick.net https://*.adtrafficquality.google https://www.googletagmanager.com https://fundingchoicesmessages.google.com https://www.gstatic.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self' https://*.googlesyndication.com https://fundingchoicesmessages.google.com; frame-src https://*.googlesyndication.com https://*.g.doubleclick.net; frame-ancestors 'self'"

      Obratite paznju: frame-ancestors se menja iz 'none' u 'self' jer Google Ads koriste iframe-ove. Takodje, 'unsafe-inline' za script-src je neophodan za AdSense jer Google ubacuje inline skripte — ali to oslabljuje XSS zastitu.

      Proverite headere odmah: Pokrenite nas skener na vasem Vercel sajtu. Za 30 sekundi cete videti tacno koji headeri nedostaju i sa kojim vrednostima.

      3. Environment varijable — Production vs Preview vs Development

      Vercel ima tri okruzenja, i svako treba da ima razlicite tajne. Evo kako sistem funkcionise:

      OkruzenjeKada se koristiPrimer env var
      ProductionDeploy sa main/master branch-aDATABASE_URL=postgres://prod-host/mydb
      PreviewSvaki PR i svaki push na ne-main branchDATABASE_URL=postgres://staging-host/mydb
      Developmentvercel dev na lokalnoj masiniDATABASE_URL=postgres://localhost/mydb

      Kritična pravila za env varijable:

      • Nikad isti API kljuc za Production i Preview — preview deploy-ovi imaju javne URL-ove. Ako koriste produkcijski Stripe kljuc, svako ko otvori preview moze pokrenuti naplate.
      • Oznacite osetljive varijable kao Sensitive — u Vercel dashboardu, kliknite ikonu oka pored varijable. Sensitive varijable se enkriptuju, ne prikazuju se u logovima i ne mogu se procitati kroz Vercel API.
      • NEXT_PUBLIC_ prefiks je opaasan — sve varijable sa ovim prefiksom se ugradjuju u JavaScript bundle i vidljive su svima koji otvore DevTools. Nikad ne stavljajte tajne u NEXT_PUBLIC_ varijable.
      • .env fajl ide u .gitignore — lokalni .env fajl je samo za development. Ako zavrsi u Git-u, tajne su zauvek u istoriji cak i ako ga obrisete.
      NEXT_PUBLIC_STRIPE_SECRET=sk_live_xxx
      Tajni kljuc sa NEXT_PUBLIC_ prefiksom — vidljiv svima u browser-u
      STRIPE_SECRET_KEY=sk_live_xxx (server-side only)
      NEXT_PUBLIC_STRIPE_PK=pk_live_xxx (samo publishable key)
      Cesta greska vibe kodera: AI genraise kod sa process.env.NEXT_PUBLIC_DATABASE_URL. Vi kopirate iz AI output-a, dodate u Vercel dashboard i ne primetite da je baza izlozena celom svetu. Uvek proverite da li varijabla zaista treba NEXT_PUBLIC_ prefiks.

      4. Preview deploy zastita

      Svaki pull request na GitHub-u automatski kreira preview deploy na Vercel-u. Ovaj deploy ima javni URL u formatu project-git-branch-username.vercel.app. Evo zasto je to rizik i kako da se zastitite:

      Problem 1: Javni pristup. Svako ko zna URL moze pristupiti. Google moze da ga indeksira. Bot-ovi mogu da ga skeniraju. Ako preview koristi produkcijski API, podaci su izlozeni.

      Problem 2: Zauvek dostupni. Svaki deploy kreira permanentan URL. Cak i kad zatvorite PR, stari deploy ostaje aktivan na unikatnom URL-u. Bagovi, test podaci i debug modovi ostaju online.

      Resenja po Vercel planu:

      • Hobby (besplatan) — koristite razlicite env varijable za Preview okruzenje. Barem ce tajne biti izolovane.
      • Pro ($20/mesec) — ukljucite Vercel Authentication (Settings → Deployment Protection). Samo ulogovani clanovi tima mogu pristupiti preview deploy-ovima.
      • Pro sa password-om — alternativno, postavite password protection za preview URL-ove (Settings → Deployment Protection → Password).
      • Enterprise — SAML SSO integracija za pristup preview deploy-ovima samo autorizovanim zaposlenima.
      Production i Preview koriste isti OPENAI_API_KEY — svako ko otvori preview URL moze trositi vas budzet
      Production: OPENAI_API_KEY=sk-prod-xxx
      Preview: OPENAI_API_KEY=sk-test-xxx (sa rate limitom od $5)

      5. Serverless i Edge funkcije — bezbednost

      Vercel izvrsava server-side kod kroz dva runtime-a: Serverless (Node.js) i Edge (V8 isolates). Oba imaju specificne bezbednosne zahteve.

      Serverless funkcije (api/ direktorijum ili Route Handlers)

      • Timeout — Hobby plan ima limit od 10 sekundi, Pro od 60 sekundi. Podesite maxDuration u vercel.json da sprecite beskonacne zahteve koji trose resurse.
      • Memorija — default je 1024 MB. Za API rute koje ne procesiraju fajlove, smanjite na 256 MB u vercel.json ("memory": 256) da ogranicite potencijalne zloupotrebe.
      • Cold start tajne — env varijable se ucitavaju pri svakom cold start-u. Nikad ih ne ispisujte u console.log() jer se logovi cuvaju i vidljivi su u Vercel dashboardu.
      • CORS — Vercel ne dodaje CORS headere automatski. Ako vasa API ruta treba da odgovara frontend-u na drugom domenu, morate rucno dodati Access-Control-Allow-Origin header.
      // vercel.json — ogranicenje za serverless funkcije { "functions": { "api/**/*.js": { "maxDuration": 15, // sekunde — dovoljno za vecinu API poziva "memory": 256 // MB — ogranicava potrosnju resursa } } }

      Edge funkcije

      • Manji napadni vektor — Edge runtime nema pristup fajl sistemu, child procesima ni Node.js modulima. To ga cini inherentno bezbednijim.
      • Globalna distribucija — izvrsava se na najblizem edge cvor-u korisniku. Idealno za autentifikaciju, redirect-e i rate limiting middleware.
      • Ogranicenja — max 128 KB velicina koda posle kompresije, nema native modula (bcrypt ne radi), ogrannicen skup Web API-ja.
      // middleware.ts — Edge middleware za rate limiting import { NextResponse } from 'next/server'; export const config = { matcher: '/api/:path*' }; export function middleware(request) { // Proverite rate limit pre nego sto zahtev stigne do serverless funkcije const ip = request.headers.get('x-forwarded-for'); // ...implementacija sa Upstash Redis ili KV store-om return NextResponse.next(); }

      6. Domen i SSL konfiguracija

      Vercel automatski izdaje SSL sertifikat za svaki domen, ali postoje dodatne stvari koje morate podesiti za potpunu zastitu:

      • Prilagodjen domen — koristite vlastiti domen umesto .vercel.app. Pored profesionalnog izgleda, vlastiti domen vam daje kontrolu nad DNS zapisima i omogucava CAA i DNSSEC.
      • CAA DNS zapis — ogranicava koji Certificate Authority sme da izda sertifikat za vas domen. Vercel koristi Let's Encrypt, pa dodajte: CAA 0 issue "letsencrypt.org". Ovo sprecava da neko drugi izda lazni sertifikat za vas domen.
      • HSTS preload lista — nakon sto ste sigurni da vam HTTPS radi korektno, podnesite vas domen na hstspreload.org. Ovo ugradjuje vas domen u Chrome/Firefox/Edge tako da se nikad ne uspostavlja HTTP konekcija, cak ni pri prvoj poseti.
      • www redirect — konfigurisite Vercel da redirektuje www.vasdomen.rs na vasdomen.rs (ili obrnuto). Konzistentan domen sprecava cookie konfuziju i pomaze SEO.
      • DNSSEC — omogucite DNSSEC kod vaseg DNS provajdera (Cloudflare ga ukljucuje jednim klikom). Stiti od DNS spoofing-a gde napadac preusmeri vas domen na maliciozan server.
      Bez CAA zapisa — bilo koji CA moze da izda sertifikat za vas domen
      Bez HSTS — prva poseta ide preko HTTP-a, podlozna MITM napadu
      CAA 0 issue "letsencrypt.org" — samo Let's Encrypt
      HSTS sa preload — browser nikad ne pokusa HTTP

      Vercel Deploy — Security Settings for Vibe Coders

      Complete guide: vercel.json headers with real values, environment variables per scope, preview deploy protection, Edge/Serverless security, domain/SSL, and deploy checklist

      1. Why Vercel requires additional configuration

      Vercel is the most popular platform for deploying Next.js, React, and static sites. AI tools like Cursor, Claude Code, and v0 generate code that deploys to Vercel with a single click. The problem is that single click does not include security configuration.

      Here is what Vercel automatically provides: SSL/TLS certificates (Let's Encrypt), global CDN with edge locations, DDoS protection at the network level, automatic builds from Git, and isolation between projects.

      And here is what Vercel does not do for you: it adds zero security HTTP headers, does not encrypt env variables by default, does not restrict access to preview deployments, and does not implement rate limiting on your API routes.

      Key distinction: Vercel gives you secure infrastructure. But configuring your application — headers, secrets, access control — is entirely your responsibility. This guide covers each of those items with exact configurations you can copy.

      2. Security headers in vercel.json — complete configuration

      This is the most important section. Vercel adds zero security headers by default. Without a vercel.json file, your site has no CSP, no HSTS, no clickjacking protection — nothing. Here is a production-ready vercel.json with all required headers:

      { "headers": [ { "source": "/(.*)", // Applies to ALL routes "headers": [ { "key": "Strict-Transport-Security", "value": "max-age=31536000; includeSubDomains; preload" }, { "key": "Content-Security-Policy", "value": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self'; frame-ancestors 'none'" }, { "key": "X-Frame-Options", "value": "DENY" }, { "key": "X-Content-Type-Options", "value": "nosniff" }, { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }, { "key": "Permissions-Policy", "value": "camera=(), microphone=(), geolocation=()" }, { "key": "Cross-Origin-Opener-Policy", "value": "same-origin" } ] } ] }

      Each header serves a specific purpose:

      • Strict-Transport-Security — tells the browser your site uses HTTPS exclusively for the next 365 days (max-age=31536000). includeSubDomains protects subdomains too. preload enables inclusion in Chrome's built-in list.
      • Content-Security-Policy — the most complex and most important header. default-src 'self' means all resources load only from your domain. Each directive (script-src, style-src, font-src) specifies exceptions. frame-ancestors 'none' replaces X-Frame-Options in modern browsers.
      • X-Frame-Options: DENY — prevents any site from loading your site in an iframe. Protects against clickjacking where an attacker places an invisible iframe over a button.
      • X-Content-Type-Options: nosniff — prevents the browser from “guessing” MIME types. Without this, a browser might interpret a text file as JavaScript.
      • Referrer-Policy — strict-origin-when-cross-origin sends the full URL only for same-origin requests; for external links it sends only the domain. Prevents leaking URL parameters (tokens, session IDs).
      • Permissions-Policy — explicitly disables access to camera, microphone, and geolocation. Even if an attacker injects a script, they cannot activate these sensors.
      • Cross-Origin-Opener-Policy — isolates your tab from other tabs. Prevents Spectre-type attacks where another tab accesses your tab's memory.

      CSP for a site with Google Ads

      If you use Google AdSense, you must expand your CSP to allow Google domains. This is a common scenario for vibe coders monetizing projects:

      "value": "default-src 'self'; script-src 'self' 'unsafe-inline' https://*.googlesyndication.com https://*.g.doubleclick.net https://*.adtrafficquality.google https://www.googletagmanager.com https://fundingchoicesmessages.google.com https://www.gstatic.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; img-src 'self' data: https:; connect-src 'self' https://*.googlesyndication.com https://fundingchoicesmessages.google.com; frame-src https://*.googlesyndication.com https://*.g.doubleclick.net; frame-ancestors 'self'"

      Note: frame-ancestors changes from 'none' to 'self' because Google Ads uses iframes. Also, 'unsafe-inline' for script-src is required by AdSense because Google injects inline scripts — but this weakens XSS protection.

      3. Environment variables — Production vs Preview vs Development

      Vercel has three environment scopes, and each should have different secrets. Here is how the system works:

      ScopeWhen usedExample env var
      ProductionDeploy from main/master branchDATABASE_URL=postgres://prod-host/mydb
      PreviewEvery PR and every push to non-main branchDATABASE_URL=postgres://staging-host/mydb
      Developmentvercel dev on local machineDATABASE_URL=postgres://localhost/mydb

      Critical rules for environment variables:

      • Never share API keys between Production and Preview — preview deploys have public URLs. If they use your production Stripe key, anyone who opens the preview can trigger real charges.
      • Mark sensitive variables as Sensitive — in the Vercel dashboard, click the eye icon next to the variable. Sensitive variables are encrypted, hidden from logs, and cannot be read via the Vercel API.
      • NEXT_PUBLIC_ prefix is dangerous — any variable with this prefix gets embedded into the JavaScript bundle and is visible to anyone who opens DevTools. Never put secrets in NEXT_PUBLIC_ variables.
      • .env file goes in .gitignore — the local .env file is for development only. If it ends up in Git, secrets are in the history forever even if you delete the file.
      NEXT_PUBLIC_STRIPE_SECRET=sk_live_xxx
      Secret key with NEXT_PUBLIC_ prefix — visible to everyone in the browser
      STRIPE_SECRET_KEY=sk_live_xxx (server-side only)
      NEXT_PUBLIC_STRIPE_PK=pk_live_xxx (publishable key only)
      Common vibe coder mistake: An AI tool generates process.env.NEXT_PUBLIC_SUPABASE_SERVICE_KEY and you copy it without noticing. The service key has admin access to your database — now it is in the JavaScript bundle visible to everyone.

      4. Preview deploy protection

      Every pull request on GitHub automatically creates a preview deploy on Vercel. This deploy has a public URL in the format project-git-branch-username.vercel.app. Here is why this is a risk and how to protect yourself:

      Problem 1: Public access. Anyone who knows the URL can access it. Google can index it. Bots can scan it. If the preview uses production APIs, your data is exposed.

      Problem 2: Permanently available. Every deploy creates a permanent URL. Even when you close the PR, the old deploy remains active on its unique URL. Bugs, test data, and debug modes stay online forever.

      Solutions by Vercel plan:

      • Hobby (free) — use different env variables for the Preview scope. At minimum, secrets will be isolated.
      • Pro ($20/month) — enable Vercel Authentication (Settings → Deployment Protection). Only logged-in team members can access preview deploys.
      • Pro with password — alternatively, set password protection for preview URLs (Settings → Deployment Protection → Password).
      • Enterprise — SAML SSO integration for preview deploy access restricted to authorized employees only.
      Production and Preview use the same OPENAI_API_KEY — anyone who opens the preview URL can spend your budget
      Production: OPENAI_API_KEY=sk-prod-xxx
      Preview: OPENAI_API_KEY=sk-test-xxx (with $5 rate limit)

      5. Serverless and Edge functions — security

      Vercel executes server-side code through two runtimes: Serverless (Node.js) and Edge (V8 isolates). Both have specific security requirements.

      Serverless functions (api/ directory or Route Handlers)

      • Timeout — Hobby plan has a 10-second limit, Pro has 60 seconds. Set maxDuration in vercel.json to prevent infinite requests that exhaust resources.
      • Memory — default is 1024 MB. For API routes that do not process files, reduce to 256 MB in vercel.json ("memory": 256) to limit potential abuse.
      • Cold start secrets — env variables load on every cold start. Never print them with console.log() because logs are stored and visible in the Vercel dashboard.
      • CORS — Vercel does not add CORS headers automatically. If your API route needs to respond to a frontend on a different domain, you must manually add the Access-Control-Allow-Origin header.
      // vercel.json — serverless function constraints { "functions": { "api/**/*.js": { "maxDuration": 15, // seconds — sufficient for most API calls "memory": 256 // MB — limits resource consumption } } }

      Edge functions

      • Smaller attack surface — Edge runtime has no access to the file system, child processes, or Node.js modules. This makes it inherently more secure.
      • Global distribution — executes on the nearest edge node to the user. Ideal for authentication, redirects, and rate limiting middleware.
      • Limitations — max 128 KB code size after compression, no native modules (bcrypt does not work), limited set of Web APIs.
      // middleware.ts — Edge middleware for rate limiting import { NextResponse } from 'next/server'; export const config = { matcher: '/api/:path*' }; export function middleware(request) { // Check rate limit before the request reaches the serverless function const ip = request.headers.get('x-forwarded-for'); // ...implementation with Upstash Redis or KV store return NextResponse.next(); }

      6. Domain and SSL configuration

      Vercel automatically issues an SSL certificate for every domain, but there are additional steps you must take for complete protection:

      • Custom domain — use your own domain instead of .vercel.app. Beyond the professional appearance, a custom domain gives you control over DNS records and enables CAA and DNSSEC.
      • CAA DNS record — restricts which Certificate Authority can issue certificates for your domain. Vercel uses Let's Encrypt, so add: CAA 0 issue "letsencrypt.org". This prevents anyone else from issuing a fraudulent certificate for your domain.
      • HSTS preload list — once you are confident HTTPS works correctly, submit your domain at hstspreload.org. This embeds your domain in Chrome/Firefox/Edge so an HTTP connection is never established, not even on the first visit.
      • www redirect — configure Vercel to redirect www.yourdomain.com to yourdomain.com (or vice versa). A consistent domain prevents cookie confusion and helps SEO.
      • DNSSEC — enable DNSSEC at your DNS provider (Cloudflare enables it with a single click). Protects against DNS spoofing where an attacker redirects your domain to a malicious server.

      7. Vercel Firewall, WAF, and rate limiting

      Vercel has multiple layers of protection at the infrastructure level, but each plan offers different capabilities:

      FeatureHobbyProEnterprise
      DDoS protection (L3/L4)YesYesYes
      Edge NetworkYesYesYes
      Vercel Firewall (rules)—YesYes

      Bot protectionBasicAdvancedAdvanced
      IP blocking—YesYes
      Geo-blocking—YesYes

      You must implement rate limiting yourself. Vercel infrastructure protects against volumetric DDoS attacks, but it does not limit the number of requests per user on your API routes. Without rate limiting, an attacker can:

      • Attempt brute-force login infinitely
      • Send thousands of requests to an API that consumes OpenAI credits
      • Flood your contact form with spam messages

      The simplest approach for vibe coders: use Upstash Redis (free tier) with the @upstash/ratelimit package in Edge middleware. Alternatively, if you use Cloudflare in front of Vercel, Cloudflare has built-in rate limiting.

      9. Most common Vercel security mistakes by vibe coders

      These are the mistakes we see in the majority of Vercel deployments that come through our scanner:

      Mistake 1: Zero headers. Vercel adds no headers by default. You create a React app, deploy it, and the site is live without CSP, without HSTS, without anything. All an attacker needs is one XSS to load an external script.

      Mistake 2: NEXT_PUBLIC_ secrets. An AI tool generates process.env.NEXT_PUBLIC_SUPABASE_SERVICE_KEY and you copy it without noticing. The service key has admin access to your database — now it is in the JavaScript bundle visible to everyone.

      Mistake 3: Same env for everything. In a rush, you set the production Stripe key for Preview too. A colleague opens a PR, Vercel creates a preview, and now a test page can process real charges.

      Mistake 4: Forgotten preview URLs. You create 50 PRs during development. Each creates a permanent URL. Old deploys with debug mode, test data, and open admin panels stay online forever.

      Mistake 5: console.log(process.env). For debugging you print all env variables to the console. The build log stores this output, visible to all team members in the Vercel dashboard.

      Mistake 6: Missing rate limiting. You deploy an AI chatbot that calls the OpenAI API. Without rate limits, a single user can exhaust your entire monthly budget in 10 minutes.
