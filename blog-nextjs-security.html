      Next.js bezbednosna checklista za vibe codere

      Konkretni primeri koda, before/after poredjenja i 12-stavki checklista za svaki Next.js projekat koji generisete sa AI alatima

      1. NEXT_PUBLIC_ zamka — greska koja otkriva vase tajne

      Ovo je greska broj jedan u Next.js projektima koje generisu AI alati. Next.js ima jednostavno pravilo: svaka env varijabla sa prefiksom NEXT_PUBLIC_ se ukljucuje u klijentski JavaScript bundle. To znaci da je vidljiva svakome ko otvori DevTools u browseru.

      Problem je sto AI alati (Cursor, Copilot, Claude) cesto dodaju NEXT_PUBLIC_ prefiks na varijable koje bi trebalo da ostanu tajne. Razlog: AI vidi da vam treba varijabla u komponenti i automatski dodaje prefiks da bi "resio" build error — ne razmisljajuci o bezbednosti.

      Opasno — tajni kljuc u browseru
      // .env.local
      NEXT_PUBLIC_SUPABASE_SERVICE_KEY=eyJhbGci...
      NEXT_PUBLIC_STRIPE_SECRET_KEY=sk_live_51...
      NEXT_PUBLIC_DATABASE_URL=postgresql://user:pass@...

      // bilo koja komponenta
      const supabase = createClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL,
        process.env.NEXT_PUBLIC_SUPABASE_SERVICE_KEY
      );
      Bezbedno — tajne samo na serveru
      // .env.local
      SUPABASE_SERVICE_KEY=eyJhbGci...
      STRIPE_SECRET_KEY=sk_live_51...
      DATABASE_URL=postgresql://user:pass@...

      // app/api/data/route.ts (server-only)
      const supabase = createClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL,
        process.env.SUPABASE_SERVICE_KEY
      );

      Pravilo: NEXT_PUBLIC_SUPABASE_URL je OK — to je javni URL. Ali SUPABASE_SERVICE_KEY nikad ne sme imati NEXT_PUBLIC_ prefiks jer daje admin pristup celoj bazi. Isto vazi za STRIPE_SECRET_KEY, DATABASE_URL, JWT_SECRET, i bilo koji API kljuc sa write pristupom.

      Kako da proverite: Otvorite sajt u browseru, pritisnite F12, idite na Sources tab i pretrazite (Ctrl+F) za "sk_live", "eyJhbG", "service_role" ili "password". Ako nadjete bilo sta — imate ozbiljan problem.

      Stvaran primer: U martu 2026, bezbednosni istrazivac je otkrio Supabase service key u klijentskom JS-u popularnog SaaS alata. Sa tim kljucem je mogao da cita, menja i brise sve podatke u bazi — ukljucujuci korisnicke email adrese i lozinke.

      2. Bezbednosni headeri u next.config.js

      Next.js ne dodaje nijedan bezbednosni HTTP header automatski. Morate ih sami konfigurisati. Bez njih, vas sajt je ranjiv na XSS, clickjacking i MIME sniffing napade.

      Evo kompletne konfiguracije za next.config.js:

      // next.config.js
      const nextConfig = {
        headers: async () => [
          {
            source: '/(.*)',
            headers: [
              { key: 'Content-Security-Policy',
                value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' https://fonts.gstatic.com; connect-src 'self' https://api.example.com" },
              { key: 'Strict-Transport-Security',
                value: 'max-age=63072000; includeSubDomains; preload' },
              { key: 'X-Frame-Options', value: 'DENY' },
              { key: 'X-Content-Type-Options', value: 'nosniff' },
              { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
              { key: 'Permissions-Policy',
                value: 'camera=(), microphone=(), geolocation=(), interest-cohort=()' },
            ]
          }
        ]
      };
      module.exports = nextConfig;

      CSP je najvazniji header. Bez njega, napadac moze da ubaci skriptu sa bilo kog domena na vas sajt. Gornji primer koristi restriktivan CSP koji dozvoljava samo resurse sa vaseg domena. Prilagodite connect-src za vase API domene i script-src ako koristite analytics ili trecu stranu.

      Ako koristite Vercel za hosting, mozete takodje konfigurisati headere u vercel.json:

      // vercel.json
      {
        "headers": [
          {
            "source": "/(.*)",
            "headers": [
              { "key": "X-Frame-Options", "value": "DENY" },
              { "key": "X-Content-Type-Options", "value": "nosniff" },
              { "key": "Strict-Transport-Security",
                "value": "max-age=63072000; includeSubDomains; preload" }
            ]
          }
        ]
      }
      Proverite vase headere: Pokrenite nas besplatni sken — proverava svih 6 kljucnih bezbednosnih headera i daje ocenu od A do F. Traje 30 sekundi.

      3. API rute — autentifikacija i validacija

      Svaka datoteka u app/api/ folderu je javno dostupan HTTP endpoint. AI alati generisu API rute koje rade, ali retko dodaju proveru ko ih poziva. Rezultat: bilo ko moze da pozove vasu API rutu direktno iz browsera ili sa curl.

      Opasno — API bez autentifikacije
      // app/api/users/route.ts
      export async function GET() {
        const users = await db.query(
          'SELECT * FROM users'
        );
        return Response.json(users);
      }
      Bezbedno — auth + validacija
      // app/api/users/route.ts
      import { auth } from '@/lib/auth';

      export async function GET(req: Request) {
        const session = await auth();
        if (!session) {
          return Response.json(
            { error: 'Unauthorized' },
            { status: 401 }
          );
        }
        const users = await db.query(
          'SELECT id, name FROM users WHERE org_id = $1',
          [session.user.orgId]
        );
        return Response.json(users);
      }

      U bezbednom primeru postoje tri kljucne razlike: (1) provera sesije na pocetku, (2) query vraca samo id i name umesto svih kolona, i (3) filtriranje po org_id tako da korisnik vidi samo svoje podatke.

      Input validacija sa Zod: Svaki POST/PUT endpoint mora da validira ulazne podatke na serveru. Frontend validacija je za UX — server validacija je za bezbednost.

      // app/api/contact/route.ts
      import { z } from 'zod';

      const ContactSchema = z.object({
        email: z.string().email(),
        message: z.string().min(10).max(1000),
        name: z.string().min(2).max(100),
      });

      export async function POST(req: Request) {
        const body = await req.json();
        const result = ContactSchema.safeParse(body);
        if (!result.success) {
          return Response.json(
            { error: 'Invalid input' },
            { status: 400 }
          );
        }
        // result.data je sada tipizirano i validirano
        await saveContact(result.data);
        return Response.json({ ok: true });
      }

      4. Server Actions — skriveni javni endpointi

      Server Actions su funkcije sa "use server" direktivom koje Next.js pretvara u HTTP POST endpointe. Svaki korisnik moze da ih pozove direktno — ne moraju da koriste vas UI. AI alati generisu Server Actions bez ikakve zastite.

      Opasno — Server Action bez zastite
      // app/actions.ts
      'use server';

      export async function deleteUser(userId: string) {
        await db.query(
          `DELETE FROM users WHERE id = '${userId}'`
        );
      }
      Bezbedno — auth + validacija + parametrizovani query
      // app/actions.ts
      'use server';
      import { auth } from '@/lib/auth';
      import { z } from 'zod';

      export async function deleteUser(userId: string) {
        const session = await auth();
        if (!session?.user.isAdmin) {
          throw new Error('Forbidden');
        }
        const id = z.string().uuid().parse(userId);
        await db.query(
          'DELETE FROM users WHERE id = $1', [id]
        );
      }

      Bezbedna verzija ima tri sloja zastite: (1) provera da je korisnik ulogovan i da je admin, (2) Zod validacija da je userId validan UUID format (sprecava SQL injection), i (3) parametrizovani query umesto string interpolacije.

      Vazno: Next.js automatski dodaje CSRF zastitu za Server Actions (ukljucuje token u formu). Ali to NE pokriva autentifikaciju ni autorizaciju — to morate sami.

      Pravilo: Svaka Server Action je API endpoint. Tretirajte je kao javni REST endpoint — jer to i jeste.

      5. Middleware auth guard

      Next.js middleware.ts se izvrsava pre svakog zahteva — pre nego sto se ucita stranica ili API ruta. To je idealno mesto za centralnu autentifikaciju. Umesto da dodajete auth proveru u svaku rutu pojedinacno, jednom je definisete u middleware-u.

      // middleware.ts (koren projekta)
      import { NextResponse } from 'next/server';
      import type { NextRequest } from 'next/server';

      const publicPaths = ['/', '/login', '/register', '/api/health'];

      export function middleware(request: NextRequest) {
        const { pathname } = request.nextUrl;
        
        // Javne rute — propusti bez provere
        if (publicPaths.some(p => pathname.startsWith(p))) {
          return NextResponse.next();
        }
        
        // Proveri session token
        const token = request.cookies.get('session-token');
        if (!token) {
          return NextResponse.redirect(
            new URL('/login', request.url)
          );
        }
        
        // Dodaj bezbednosne headere na svaki response
        const response = NextResponse.next();
        response.headers.set('X-Frame-Options', 'DENY');
        response.headers.set('X-Content-Type-Options', 'nosniff');
        return response;
      }

      export const config = {
        // Primeni na sve rute osim statickih fajlova
        matcher: ['/((?!_next/static|_next/image|favicon.ico).*)']
      };

      Matcher je kriticno podesavanje. Bez matcher-a, middleware se izvrsava i za staticke fajlove (slike, CSS, JS), sto usporava sajt. Gornji regex iskljucuje Next.js interne putanje.

      Middleware takodje moze da dodaje bezbednosne headere dinamicki — korisno kad header zavisi od rute ili korisnika (npr. razlicit CSP za admin panel).

      6. CORS u Route Handler-ima

      Next.js ne konfigurise CORS automatski. Ako vam frontend poziva API sa drugog domena (ili cak sa drugog porta u development-u), dobicete CORS gresku. AI alati cesto "rese" ovo dodavanjem wildcard CORS-a — sto je opasno.

      Opasno — wildcard CORS
      // app/api/data/route.ts
      export async function GET() {
        return Response.json(data, {
          headers: {
            'Access-Control-Allow-Origin': '*',
            'Access-Control-Allow-Methods': '*',
          }
        });
      }
      Bezbedno — eksplicitni origin
      // app/api/data/route.ts
      const ALLOWED = [
        'https://myapp.com',
        'https://admin.myapp.com',
      ];

      export async function GET(req: Request) {
        const origin = req.headers.get('origin');
        const headers: Record<string, string> = {};
        if (origin && ALLOWED.includes(origin)) {
          headers['Access-Control-Allow-Origin'] = origin;
          headers['Access-Control-Allow-Methods'] = 'GET, POST';
        }
        return Response.json(data, { headers });
      }

      Wildcard (*) znaci da BILO KOJI sajt moze da poziva vase API rute. Napadac moze da napravi lazni sajt koji poziva vase API-je iz korisnikovog browsera, koristeci korisnikove kolacice. Sa eksplicitnim origin listom, samo vasi domeni mogu da pozivaju API.

      7. Error handling — ne otkrivajte stack trace

      Kad se desi greska u API ruti, Next.js u development modu prikazuje kompletan stack trace sa putanjama fajlova, verzijama paketa i ponekad env varijablama. Ako ostavite ovo u produkciji, napadac dobija mapu vaseg projekta.

      Opasno — detaljna greska u produkciji
      export async function POST(req: Request) {
        try {
          const data = await processOrder(req);
          return Response.json(data);
        } catch (err) {
          return Response.json(
            { error: err.message, stack: err.stack },
            { status: 500 }
          );
        }
      }
      Bezbedno — loguj interno, vrati genericki odgovor
      export async function POST(req: Request) {
        try {
          const data = await processOrder(req);
          return Response.json(data);
        } catch (err) {
          console.error('Order failed:', err);
          return Response.json(
            { error: 'Internal server error' },
            { status: 500 }
          );
        }
      }

      9. Najcesce greske vibe codera u Next.js

      Ovo su greske koje vidimo najcesce u Next.js projektima napravljenim sa AI alatima:

      • Izlozene tajne kroz NEXT_PUBLIC_ — AI dodaje prefiks da resi build error, ne razmisljajuci da tajni kljuc postaje javan. Proverite svaki .env fajl pre deploy-a.
      • Nezasticene API rute — AI generise handler koji radi, ali ne dodaje auth. Svaki endpoint bez autentifikacije je otvoren ceo internet.
      • Server Actions bez validacije — AI koristi "use server" direktivu ali ne dodaje Zod validaciju. Korisnik moze da posalje bilo sta.
      • Wildcard CORS — AI dodaje Access-Control-Allow-Origin: * da resi CORS gresku u development-u, a vi zaboravite da promenite za produkciju.
      • Nedostaju bezbednosni headeri — next.config.js generisan od AI-a gotovo nikad ne ukljucuje headers() konfiguraciju. Morate je sami dodati.
      • Debug mod u produkciji — detaljne greske sa stack trace-om vidljive u produkciji otkrivaju strukturu vaseg projekta napadacima.

      Svaku od ovih gresaka mozete otkriti automatski pokretanjem naseg skenera posle svakog deploy-a.

      1. The NEXT_PUBLIC_ trap — the mistake that exposes your secrets

      This is the number one mistake in AI-generated Next.js projects. Next.js has a simple rule: any env variable prefixed with NEXT_PUBLIC_ is bundled into the client-side JavaScript. That means it is visible to anyone who opens DevTools in their browser.

      The problem is that AI tools (Cursor, Copilot, Claude) frequently add the NEXT_PUBLIC_ prefix to variables that should remain secret. The reason: AI sees you need a variable in a component and automatically adds the prefix to "solve" the build error — without considering security.

      Dangerous — secret key in the browser
      // .env.local
      NEXT_PUBLIC_SUPABASE_SERVICE_KEY=eyJhbGci...
      NEXT_PUBLIC_STRIPE_SECRET_KEY=sk_live_51...
      NEXT_PUBLIC_DATABASE_URL=postgresql://user:pass@...

      // any component
      const supabase = createClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL,
        process.env.NEXT_PUBLIC_SUPABASE_SERVICE_KEY
      );
      Safe — secrets on the server only
      // .env.local
      SUPABASE_SERVICE_KEY=eyJhbGci...
      STRIPE_SECRET_KEY=sk_live_51...
      DATABASE_URL=postgresql://user:pass@...

      // app/api/data/route.ts (server-only)
      const supabase = createClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL,
        process.env.SUPABASE_SERVICE_KEY
      );

      The rule: NEXT_PUBLIC_SUPABASE_URL is fine — it is a public URL. But SUPABASE_SERVICE_KEY must never have the NEXT_PUBLIC_ prefix because it grants admin access to the entire database. The same applies to STRIPE_SECRET_KEY, DATABASE_URL, JWT_SECRET, and any API key with write access.

      How to check: Open your site in the browser, press F12, go to the Sources tab and search (Ctrl+F) for "sk_live", "eyJhbG", "service_role", or "password". If you find anything — you have a serious problem.

      Real-world example: In March 2026, a security researcher discovered a Supabase service key in the client-side JS of a popular SaaS tool. With that key, they could read, modify, and delete all data in the database — including user email addresses and passwords.

      2. Security headers in next.config.js

      Next.js does not add any security HTTP headers automatically. You must configure them yourself. Without them, your site is vulnerable to XSS, clickjacking, and MIME sniffing attacks.

      Here is the complete configuration for next.config.js:

      // next.config.js
      const nextConfig = {
        headers: async () => [
          {
            source: '/(.*)',
            headers: [
              { key: 'Content-Security-Policy',
                value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' https://fonts.gstatic.com; connect-src 'self' https://api.example.com" },
              { key: 'Strict-Transport-Security',
                value: 'max-age=63072000; includeSubDomains; preload' },
              { key: 'X-Frame-Options', value: 'DENY' },
              { key: 'X-Content-Type-Options', value: 'nosniff' },
              { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
              { key: 'Permissions-Policy',
                value: 'camera=(), microphone=(), geolocation=(), interest-cohort=()' },
            ]
          }
        ]
      };
      module.exports = nextConfig;

      CSP is the most important header. Without it, an attacker can inject a script from any domain onto your site. The example above uses a restrictive CSP that only allows resources from your domain. Adjust connect-src for your API domains and script-src if you use analytics or third parties.

      If you use Vercel for hosting, you can also configure headers in vercel.json:

      // vercel.json
      {
        "headers": [
          {
            "source": "/(.*)",
            "headers": [
              { "key": "X-Frame-Options", "value": "DENY" },
              { "key": "X-Content-Type-Options", "value": "nosniff" },
              { "key": "Strict-Transport-Security",
                "value": "max-age=63072000; includeSubDomains; preload" }
            ]
          }
        ]
      }

      3. API routes — authentication and validation

      Every file in the app/api/ folder is a publicly accessible HTTP endpoint. AI tools generate API routes that work, but rarely add checks for who is calling them. The result: anyone can call your API route directly from the browser or with curl.

      Dangerous — API without authentication
      // app/api/users/route.ts
      export async function GET() {
        const users = await db.query(
          'SELECT * FROM users'
        );
        return Response.json(users);
      }
      Safe — auth + validation
      // app/api/users/route.ts
      import { auth } from '@/lib/auth';

      export async function GET(req: Request) {
        const session = await auth();
        if (!session) {
          return Response.json(
            { error: 'Unauthorized' },
            { status: 401 }
          );
        }
        const users = await db.query(
          'SELECT id, name FROM users WHERE org_id = $1',
          [session.user.orgId]
        );
        return Response.json(users);
      }

      The safe version has three key differences: (1) session check at the start, (2) the query returns only id and name instead of all columns, and (3) filtering by org_id so users only see their own data.

      Input validation with Zod: Every POST/PUT endpoint must validate input on the server. Frontend validation is for UX — server validation is for security.

      // app/api/contact/route.ts
      import { z } from 'zod';

      const ContactSchema = z.object({
        email: z.string().email(),
        message: z.string().min(10).max(1000),
        name: z.string().min(2).max(100),
      });

      export async function POST(req: Request) {
        const body = await req.json();
        const result = ContactSchema.safeParse(body);
        if (!result.success) {
          return Response.json(
            { error: 'Invalid input' },
            { status: 400 }
          );
        }
        // result.data is now typed and validated
        await saveContact(result.data);
        return Response.json({ ok: true });
      }

      4. Server Actions — hidden public endpoints

      Server Actions are functions with the "use server" directive that Next.js turns into HTTP POST endpoints. Any user can call them directly — they do not need to use your UI. AI tools generate Server Actions without any protection.

      Dangerous — unprotected Server Action
      // app/actions.ts
      'use server';

      export async function deleteUser(userId: string) {
        await db.query(
          `DELETE FROM users WHERE id = '${userId}'`
        );
      }
      Safe — auth + validation + parameterized query
      // app/actions.ts
      'use server';
      import { auth } from '@/lib/auth';
      import { z } from 'zod';

      export async function deleteUser(userId: string) {
        const session = await auth();
        if (!session?.user.isAdmin) {
          throw new Error('Forbidden');
        }
        const id = z.string().uuid().parse(userId);
        await db.query(
          'DELETE FROM users WHERE id = $1', [id]
        );
      }

      The safe version has three layers of protection: (1) check that the user is logged in and is an admin, (2) Zod validation that userId is a valid UUID format (prevents SQL injection), and (3) parameterized query instead of string interpolation.

      Important: Next.js automatically adds CSRF protection for Server Actions (includes a token in the form). But that does NOT cover authentication or authorization — you must add those yourself.

      Rule of thumb: Every Server Action is an API endpoint. Treat it as a public REST endpoint — because that is what it is.

      5. Middleware auth guard

      Next.js middleware.ts runs before every request — before any page or API route loads. That makes it the ideal place for centralized authentication. Instead of adding auth checks to every route individually, define it once in middleware.

      // middleware.ts (project root)
      import { NextResponse } from 'next/server';
      import type { NextRequest } from 'next/server';

      const publicPaths = ['/', '/login', '/register', '/api/health'];

      export function middleware(request: NextRequest) {
        const { pathname } = request.nextUrl;
        
        // Public routes — pass through without checking
        if (publicPaths.some(p => pathname.startsWith(p))) {
          return NextResponse.next();
        }
        
        // Check session token
        const token = request.cookies.get('session-token');
        if (!token) {
          return NextResponse.redirect(
            new URL('/login', request.url)
          );
        }
        
        // Add security headers to every response
        const response = NextResponse.next();
        response.headers.set('X-Frame-Options', 'DENY');
        response.headers.set('X-Content-Type-Options', 'nosniff');
        return response;
      }

      export const config = {
        // Apply to all routes except static files
        matcher: ['/((?!_next/static|_next/image|favicon.ico).*)']
      };

      The matcher is a critical setting. Without it, middleware runs for static files too (images, CSS, JS), which slows down the site. The regex above excludes Next.js internal paths.

      Middleware can also add security headers dynamically — useful when a header depends on the route or user (e.g., different CSP for the admin panel).

      6. CORS in Route Handlers

      Next.js does not configure CORS automatically. If your frontend calls an API from another domain (or even another port during development), you will get a CORS error. AI tools often "solve" this by adding wildcard CORS — which is dangerous.

      Dangerous — wildcard CORS
      // app/api/data/route.ts
      export async function GET() {
        return Response.json(data, {
          headers: {
            'Access-Control-Allow-Origin': '*',
            'Access-Control-Allow-Methods': '*',
          }
        });
      }
      Safe — explicit origin allowlist
      // app/api/data/route.ts
      const ALLOWED = [
        'https://myapp.com',
        'https://admin.myapp.com',
      ];

      export async function GET(req: Request) {
        const origin = req.headers.get('origin');
        const headers: Record<string, string> = {};
        if (origin && ALLOWED.includes(origin)) {
          headers['Access-Control-Allow-Origin'] = origin;
          headers['Access-Control-Allow-Methods'] = 'GET, POST';
        }
        return Response.json(data, { headers });
      }

      A wildcard (*) means ANY website can call your API routes. An attacker can create a fake site that calls your APIs from the user's browser, using the user's cookies. With an explicit origin list, only your domains can call the API.

      7. Error handling — do not expose stack traces

      When an error occurs in an API route, Next.js in development mode shows the full stack trace with file paths, package versions, and sometimes env variables. If you leave this in production, an attacker gets a map of your project.

      Dangerous — detailed error in production
      export async function POST(req: Request) {
        try {
          const data = await processOrder(req);
          return Response.json(data);
        } catch (err) {
          return Response.json(
            { error: err.message, stack: err.stack },
            { status: 500 }
          );
        }
      }
      Safe — log internally, return generic response
      export async function POST(req: Request) {
        try {
          const data = await processOrder(req);
          return Response.json(data);
        } catch (err) {
          console.error('Order failed:', err);
          return Response.json(
            { error: 'Internal server error' },
            { status: 500 }
          );
        }
      }

      9. Most common vibe coder mistakes in Next.js

      These are the mistakes we see most frequently in AI-generated Next.js projects:

      • Exposing secrets via NEXT_PUBLIC_ — AI adds the prefix to solve a build error, without considering that the secret key becomes public. Check every .env file before deploying.
      • Unprotected API routes — AI generates handlers that work but does not add auth. Every endpoint without authentication is open to the entire internet.
      • Server Actions without validation — AI uses the "use server" directive but does not add Zod validation. Users can send anything.
      • Wildcard CORS — AI adds Access-Control-Allow-Origin: * to fix a CORS error during development, and you forget to change it for production.
      • Missing security headers — AI-generated next.config.js almost never includes the headers() configuration. You must add it yourself.
      • Debug mode in production — detailed errors with stack traces visible in production reveal your project structure to attackers.
