      1. Zasto napadaci ciljaju WordPress?

      WordPress pokrece 43% svih web sajtova na internetu — to je skoro polovina celog weba. Ova dominacija ga cini najatraktivnijom metom za automatizovane napade iz prostog razloga: jedna ranjivost u popularnom pluginu potencijalno pogadja milione sajtova istovremeno.

      Vecina WordPress napada nije ciljana. Automatski botovi neprekidno skeniraju internet trazeci poznate ranjivosti u popularnim pluginima i temama. Kad pronadju ranjiv sajt, eksploatacija je automatska — ne zahteva ljudsku intervenciju.

      Dodatni faktori koji doprinose:

      • Ogroman ekosistem plugina — preko 60.000 plugina u zvanicnom repozitorijumu, od kojih mnogi imaju neujednacen kvalitet koda i bezbednosne prakse.
      • Korisnici bez tehnickog znanja — WordPress privlaci korisnike koji nisu programeri, pa cesto zanemaruju azuriranja i bezbednosna podesavanja.
      • Podrazumevana konfiguracija — sveza WordPress instalacija ima predvidive putanje (/wp-admin, /wp-login.php, /xmlrpc.php), poznate database tabele (wp_users, wp_posts), i otkrivenu verziju.
      • Shared hosting — mnogi WP sajtovi su na jeftinom deljenom hostingu gde kompromitovanje jednog sajta moze ugroziti sve ostale na istom serveru.
      Statistika (Sucuri 2025): WordPress cini preko 90% svih hakovanih CMS sajtova. 97% napada dolazi kroz ranjive plugine, ne kroz WordPress core koji je sam po sebi relativno bezbedan.

      2. Najcesci vektori napada na WordPress

      Razumevanje nacina na koje se WP sajtovi kompromituju je kljucno za efikasnu odbranu. Evo sest najcescih vektora:

      2.1. Zastareli plugini i teme

      Ovo je uzrok broj 1 svih WordPress hakovanja. Plugini sa poznatim ranjivostima ostaju instalirani i neazurirani mesecima ili godinama. Botovi koriste javno objavljene CVE-ove (Common Vulnerabilities and Exposures) da automatski eksploatisu ove ranjivosti u roku od sati nakon objavljivanja zakrpe.

      Posebno su opasni plugini koji su napusteni od strane autora — nemaju vise bezbednosne ispravke, ali ostaju instalirani na hiljadama sajtova.

      Zastita: Redovno azurirajte sve plugine i teme. Uklonite plugine koje ne koristite. Omogucite auto-update za manje plugine. Proveravajte da li plugin ima aktivnog autora pre instalacije.

      2.2. Slabe lozinke i brute-force

      WordPress login stranica (/wp-login.php) je podrazumevano dostupna svima. Bez rate limiting-a, napadaci mogu da probaju hiljade lozinki u minutu. Cesti targetirani korisnici: "admin", "administrator", ime domena.

      Zastita: Koristite jake lozinke (minimum 16 karaktera sa mesavinom slova, brojeva i specijalnih znakova), omogucite dvofaktorsku autentifikaciju (2FA), ogranicite pristup login stranici, i implementirajte rate limiting ili CAPTCHA nakon nekoliko neuspelih pokusaja.

      2.3. File inclusion ranjivosti (LFI/RFI)

      Mnogi plugini ucitavaju fajlove bez pravilne validacije korisnickog unosa, sto moze omoguciti ucitavanje neovlascenih fajlova sa servera (Local File Inclusion) ili sa eksternog izvora (Remote File Inclusion).

      Zastita: Koristite samo proverene plugine sa aktivnim odrzavanjem i velikim brojem instalacija. Nas skener proverava pristupacnost osetljivih fajlova i direktorijuma.

      2.4. SQL Injection

      Plugini koji ne koriste WordPress-ove ugradjene funkcije za rad sa bazom ($wpdb->prepare()) su potencijalno ranjivi. Ovo je posebno cest problem kod plugina koji imaju sopstvene forme za pretragu ili filtriranje podataka.

      Zastita: Koristite samo plugine koji prate WordPress Coding Standards. Proverite da li plugin koristi pripremljene upite (prepared statements). Redovno skenirajte sajt.

      2.5. Cross-Site Scripting (XSS)

      XSS ranjivosti nastaju kada sajt prikazuje korisnicki unos bez pravilnog filtriranja. Napadac moze ubaciti JavaScript kod koji se izvrsava u browseru posetilaca, kradeci kolacice ili preusmeravajuci korisnike.

      Zastita: Konfigurisite Content Security Policy (CSP) header. Koristite WordPress-ove esc_html(), esc_attr(), wp_kses() funkcije u temama i pluginima. Nas skener proverava CSP konfiguraciju.

      2.6. Nedostajuci bezbednosni headeri

      Vecina WordPress sajtova nema pravilno konfigurisane HTTP bezbednosne headere. Ovo ih ostavlja ranjivim na razlicite napade:

      Bez headera:
      • Nema CSP → XSS rizik
      • Nema HSTS → Downgrade rizik
      • Nema X-Frame-Options → Clickjacking
      • Nema Referrer-Policy → Curenje URL-ova
      Sa headerima:
      • CSP blokira neovlascene skripte
      • HSTS forsira HTTPS
      • X-Frame-Options brani od framinga
      • Referrer-Policy kontrolise podatke

      Zastita: Konfigurisite headere kroz .htaccess, nginx config, ili bezbednosni plugin. Pokrenite nas skener da vidite sta nedostaje.

      3. Ojacavanje wp-config.php

      Fajl wp-config.php je srce WordPress konfiguracije. Pravilno podesavanje ovog fajla dramaticno smanjuje povrsinu napada. Evo najvaznijih izmena:

      wp-config.php — bezbednosno ojacavanje // 1. Onemogucite editovanje fajlova iz admin panela define('DISALLOW_FILE_EDIT', true); // 2. Onemogucite instalaciju/azuriranje plugina iz admin panela // (koristite ovo samo ako azurirate rucno preko FTP/SSH) define('DISALLOW_FILE_MODS', true); // 3. Forsirajte SSL za admin panel define('FORCE_SSL_ADMIN', true); // 4. Ogranicite revizije postova (smanjuje velicinu baze) define('WP_POST_REVISIONS', 5); // 5. Iskljucite debug mode na produkciji define('WP_DEBUG', false); define('WP_DEBUG_LOG', false); define('WP_DEBUG_DISPLAY', false); // 6. Promenite database prefix (radite ovo SAMO pri novoj instalaciji) // Umesto podrazumevanog 'wp_' koristite nesto unikatno: $table_prefix = 'x9k2_'; // 7. Automatsko praznjenje korpe define('EMPTY_TRASH_DAYS', 7); // 8. Unikatni bezbednosni kljucevi (generisi na: api.wordpress.org/secret-key) define('AUTH_KEY', 'vas-unikatni-kljuc-ovde'); define('SECURE_AUTH_KEY', 'vas-unikatni-kljuc-ovde'); define('LOGGED_IN_KEY', 'vas-unikatni-kljuc-ovde'); define('NONCE_KEY', 'vas-unikatni-kljuc-ovde');
      Vazno: Nikada ne menjajte database prefix na vec postojecem sajtu bez prethodnog backup-a i migracije svih tabela. Ova promena je bezbedna samo pri novoj instalaciji.

      4. Bezbednosna konfiguracija .htaccess

      Za Apache servere, .htaccess fajl je mocan alat za bezbednosno ojacavanje. Evo najvaznijih pravila:

      .htaccess — bezbednosna pravila # Zastita wp-config.php od direktnog pristupa <Files wp-config.php> Order Allow,Deny Deny from all </Files> # Onemogucite directory listing Options -Indexes # Blokirajte pristup .htaccess fajlu <Files .htaccess> Order Allow,Deny Deny from all </Files> # Onemogucite XML-RPC (ako ga ne koristite) <Files xmlrpc.php> Order Allow,Deny Deny from all </Files> # Bezbednosni headeri <IfModule mod_headers.c> Header set X-Content-Type-Options "nosniff" Header set X-Frame-Options "SAMEORIGIN" Header set X-XSS-Protection "1; mode=block" Header set Referrer-Policy "strict-origin-when-cross-origin" Header set Permissions-Policy "geolocation=(), camera=(), microphone=()" Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains" </IfModule> # Zastita uploads direktorijuma od izvrsavanja PHP-a <Directory "wp-content/uploads"> <Files "*.php"> Order Allow,Deny Deny from all </Files> </Directory> # Sprecite pristup osetljivim fajlovima <FilesMatch "(^\.htaccess|\.htpasswd|wp-config\.php|readme\.html|license\.txt|error_log|debug\.log)$"> Order Allow,Deny Deny from all </FilesMatch>
      Napomena: Testirajte svako pravilo pojedinacno nakon dodavanja. Pogresna .htaccess konfiguracija moze uciniti sajt nedostupnim. Uvek imajte backup prethodne verzije.

      5. Sta nas skener proverava na WordPress-u

      Nas Web Security Scanner automatski detektuje da sajt koristi WordPress i izvrsava specijalizovane provere:

      • SSL/TLS konfiguracija — validan sertifikat, jaki cipher suite-ovi, HTTPS redirect, HSTS header.
      • Bezbednosni headeri — CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy, CORP, COEP.
      • Osetljivi fajlovi — wp-config.php backup, debug.log, .htaccess, xmlrpc.php, readme.html (otkrivanje verzije), install.php, wp-cron.php dostupnost.
      • Otvoreni portovi — provera 10+ kriticnih portova (FTP, SSH, MySQL, phpMyAdmin) koji ne bi trebali biti javno dostupni.
      • DNS bezbednost — SPF, DMARC, DNSSEC konfiguracija za zastitu od email spoofing-a.
      • Verzija softvera — detekcija WordPress verzije, otkrivanje poznatih CVE-ova za tu verziju.
      • CORS konfiguracija — da li je Cross-Origin Resource Sharing pravilno podesen i ne dozvoljava neovlascene domene.
      • Cookie bezbednost — Secure, HttpOnly, SameSite atributi na svim kolacicima.
      • Mixed content — detekcija HTTP resursa na HTTPS stranicama.
      • Informaciono curenje — PHP verzija u headerima, WordPress generator meta tag, detaljne poruke o greskama.
      Besplatno skeniranje: Unesite URL vaseg WordPress sajta i dobijte kompletni izvestaj za 30 sekundi. Bez registracije, bez modifikacije vaseg sajta, potpuno neinvazivno.

      8. AI-generisane teme i plugini — dodatni rizici

      Sve vise WordPress korisnika koristi AI alate (ChatGPT, Claude, Copilot) za generisanje tema i plugina. Ovo uvodi specificne bezbednosne rizike:

      • Nedostatak sanitizacije unosa — AI-generisan kod cesto ne koristi WordPress-ove bezbednosne funkcije poput sanitize_text_field(), wp_nonce_field(), i check_admin_referer().
      • Direktni SQL upiti — AI cesto generiSe direktne SQL upite umesto koriscenja $wpdb->prepare(), sto otvara vrata za SQL injection.
      • Nesigurno rukovanje fajlovima — Upload funkcionalnost generisana AI-jem cesto preskace MIME type verifikaciju i zastitu od path traversal napada.
      • Zastareli obrasci — AI modeli su trenirani na kodu koji moze koristiti zastarele WordPress API-je sa poznatim ranjivostima.
      • Nedostatak capability provera — AI-generisani admin paneli cesto ne proveravaju current_user_can() pre izvrsavanja privilegovanih operacija.
      AI-generisan kod bez provere:
      $_GET['id'] direktno u SQL upitu
      echo $_POST['name'] bez escape-a
      Nema nonce verifikacije u formama
      Pravilan WordPress kod:
      $wpdb->prepare() za sve upite
      esc_html() za svaki ispis
      wp_verify_nonce() na svim formama

      Preporuka: Ako koristite AI za generisanje WordPress koda, uvek pregledajte generisan kod sa fokusom na bezbednost. Pokrenite nas bezbednosni skener nakon svake promene.

      9. Sta ako je sajt vec hakovan?

      Ako sumnjate da je vas WordPress sajt kompromitovan, sledite ove korake redom:

      1. Ne panicite, ali reagujte brzo — svaki sat koji prodje povecava stetu i rizik da Google stavi sajt na crnu listu.
      2. Izolujte sajt — stavite sajt u maintenance mode ili ga privremeno ugasite da zastite posetioce od malicioznog koda.
      3. Promenite sve lozinke — WordPress admin, FTP, cPanel, baza podataka, email nalozi vezani za sajt.
      4. Skenirajte za malware — koristite nas malware skener za detekciju zlonamernog koda, skrivenih backdoor-ova i sumnjivih fajlova.
      5. Proverite admin naloge — potrazite nepoznate administratorske naloge koji su mozda kreirani tokom napada.
      6. Ocistite sajt — uklonite maliciozni kod, obrisite sumnjive fajlove, ponovo instalirajte cist WordPress core.
      7. Azurirajte sve — WordPress core, svi plugini, sve teme. Uklonite nekoriscene plugine i teme.
      8. Ojacajte konfiguraciju — primenite sve preporuke iz sekcija 3 i 4 ovog vodicA.
      9. Ponovo skenirajte — potvrdite da je sajt cist pre nego sto ga vratite online.
      10. Pratite sajt — pojacajte monitoring u narednih 30 dana jer se napadaci cesto vracaju.

      1. Why do attackers target WordPress?

      WordPress powers 43% of all websites on the internet — that is nearly half the web. This dominance makes it the most attractive target for automated attacks for a simple reason: one vulnerability in a popular plugin potentially affects millions of sites simultaneously.

      Most WordPress attacks are not targeted. Automated bots continuously scan the internet looking for known vulnerabilities in popular plugins and themes. When they find a vulnerable site, exploitation is automatic — no human intervention required.

      Additional contributing factors:

      • Massive plugin ecosystem — over 60,000 plugins in the official repository, many with inconsistent code quality and security practices.
      • Non-technical users — WordPress attracts users who are not developers, so they often neglect updates and security settings.
      • Default configuration — a fresh WordPress installation has predictable paths (/wp-admin, /wp-login.php, /xmlrpc.php), known database tables (wp_users, wp_posts), and an exposed version number.
      • Shared hosting — many WP sites run on cheap shared hosting where compromising one site can jeopardize all others on the same server.
      Statistic (Sucuri 2025): WordPress accounts for over 90% of all hacked CMS sites. 97% of attacks come through vulnerable plugins, not through WordPress core, which is itself relatively secure.

      2. Most common WordPress attack vectors

      Understanding how WP sites get compromised is key to effective defense. Here are the six most common vectors:

      2.1. Outdated plugins and themes

      This is cause number one of all WordPress hacks. Plugins with known vulnerabilities remain installed and unpatched for months or years. Bots use publicly disclosed CVEs (Common Vulnerabilities and Exposures) to automatically exploit these vulnerabilities within hours of a patch being released.

      Particularly dangerous are plugins abandoned by their authors — they no longer receive security fixes but remain installed on thousands of sites.

      Protection: Regularly update all plugins and themes. Remove plugins you do not use. Enable auto-update for minor plugins. Check if a plugin has an active author before installing.

      2.2. Weak passwords and brute-force

      The WordPress login page (/wp-login.php) is accessible to everyone by default. Without rate limiting, attackers can try thousands of passwords per minute. Commonly targeted usernames: "admin", "administrator", the domain name.

      Protection: Use strong passwords (minimum 16 characters with a mix of letters, numbers, and special characters), enable two-factor authentication (2FA), restrict access to the login page, and implement rate limiting or CAPTCHA after several failed attempts.

      2.3. File inclusion vulnerabilities (LFI/RFI)

      Many plugins load files without proper input validation, which can allow loading unauthorized files from the server (Local File Inclusion) or from an external source (Remote File Inclusion).

      Protection: Only use verified plugins with active maintenance and a large number of installations. Our scanner checks accessibility of sensitive files and directories.

      2.4. SQL Injection

      Plugins that do not use WordPress built-in database functions ($wpdb->prepare()) are potentially vulnerable. This is especially common in plugins that have their own search or data filtering forms.

      Protection: Only use plugins that follow WordPress Coding Standards. Verify that the plugin uses prepared statements. Scan your site regularly.

      2.5. Cross-Site Scripting (XSS)

      XSS vulnerabilities occur when a site displays user input without proper escaping. An attacker can inject JavaScript code that executes in visitors' browsers, stealing cookies or redirecting users.

      Protection: Configure Content Security Policy (CSP) header. Use WordPress functions like esc_html(), esc_attr(), wp_kses() in themes and plugins. Our scanner checks CSP configuration.

      2.6. Missing security headers

      Most WordPress sites lack properly configured HTTP security headers. This leaves them vulnerable to various attacks:

      Without headers:
      • No CSP → XSS risk
      • No HSTS → Downgrade risk
      • No X-Frame-Options → Clickjacking
      • No Referrer-Policy → URL leakage
      With headers:
      • CSP blocks unauthorized scripts
      • HSTS enforces HTTPS
      • X-Frame-Options prevents framing
      • Referrer-Policy controls data flow

      Protection: Configure headers through .htaccess, nginx config, or a security plugin. Run our scanner to see what is missing.

      3. Hardening wp-config.php

      The wp-config.php file is the heart of WordPress configuration. Properly configuring this file dramatically reduces the attack surface. Here are the most important changes:

      wp-config.php — security hardening // 1. Disable file editing from the admin panel define('DISALLOW_FILE_EDIT', true); // 2. Disable plugin/theme installation from admin panel // (use this only if you update manually via FTP/SSH) define('DISALLOW_FILE_MODS', true); // 3. Force SSL for the admin panel define('FORCE_SSL_ADMIN', true); // 4. Limit post revisions (reduces database size) define('WP_POST_REVISIONS', 5); // 5. Disable debug mode in production define('WP_DEBUG', false); define('WP_DEBUG_LOG', false); define('WP_DEBUG_DISPLAY', false); // 6. Change database prefix (do this ONLY on new installations) // Instead of the default 'wp_' use something unique: $table_prefix = 'x9k2_'; // 7. Auto-empty trash define('EMPTY_TRASH_DAYS', 7); // 8. Unique security keys (generate at: api.wordpress.org/secret-key) define('AUTH_KEY', 'your-unique-key-here'); define('SECURE_AUTH_KEY', 'your-unique-key-here'); define('LOGGED_IN_KEY', 'your-unique-key-here'); define('NONCE_KEY', 'your-unique-key-here');
      Important: Never change the database prefix on an existing site without a prior backup and full table migration. This change is only safe on new installations.

      4. Security configuration with .htaccess

      For Apache servers, the .htaccess file is a powerful tool for security hardening. Here are the most important rules:

      .htaccess — security rules # Protect wp-config.php from direct access <Files wp-config.php> Order Allow,Deny Deny from all </Files> # Disable directory listing Options -Indexes # Block access to .htaccess itself <Files .htaccess> Order Allow,Deny Deny from all </Files> # Disable XML-RPC (if you do not use it) <Files xmlrpc.php> Order Allow,Deny Deny from all </Files> # Security headers <IfModule mod_headers.c> Header set X-Content-Type-Options "nosniff" Header set X-Frame-Options "SAMEORIGIN" Header set X-XSS-Protection "1; mode=block" Header set Referrer-Policy "strict-origin-when-cross-origin" Header set Permissions-Policy "geolocation=(), camera=(), microphone=()" Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains" </IfModule> # Protect uploads directory from PHP execution <Directory "wp-content/uploads"> <Files "*.php"> Order Allow,Deny Deny from all </Files> </Directory> # Prevent access to sensitive files <FilesMatch "(^\.htaccess|\.htpasswd|wp-config\.php|readme\.html|license\.txt|error_log|debug\.log)$"> Order Allow,Deny Deny from all </FilesMatch>
      Note: Test each rule individually after adding it. An incorrect .htaccess configuration can make your site inaccessible. Always keep a backup of the previous version.

      8. AI-generated themes and plugins — additional risks

      More and more WordPress users are using AI tools (ChatGPT, Claude, Copilot) to generate themes and plugins. This introduces specific security risks:

      • Missing input sanitization — AI-generated code often does not use WordPress security functions like sanitize_text_field(), wp_nonce_field(), and check_admin_referer().
      • Direct SQL queries — AI frequently generates direct SQL queries instead of using $wpdb->prepare(), opening the door to SQL injection.
      • Insecure file handling — Upload functionality generated by AI often skips MIME type verification and path traversal protection.
      • Outdated patterns — AI models are trained on code that may use deprecated WordPress APIs with known vulnerabilities.
      • Missing capability checks — AI-generated admin panels often do not check current_user_can() before executing privileged operations.
      AI-generated code without checks:
      $_GET['id'] directly in SQL query
      echo $_POST['name'] without escaping
      No nonce verification on forms
      Proper WordPress code:
      $wpdb->prepare() for all queries
      esc_html() for every output
      wp_verify_nonce() on all forms

      Recommendation: If you use AI to generate WordPress code, always review the generated code with a focus on security. Run our security scanner after every change.

      9. What if the site is already hacked?

      If you suspect your WordPress site is compromised, follow these steps in order:

      1. Do not panic, but act quickly — every hour that passes increases the damage and the risk of Google blacklisting your site.
      2. Isolate the site — put the site in maintenance mode or temporarily take it down to protect visitors from malicious code.
      3. Change all passwords — WordPress admin, FTP, cPanel, database, email accounts linked to the site.
      4. Scan for malware — use our malware scanner for detection of malicious code, hidden backdoors, and suspicious files.
      5. Check admin accounts — look for unknown administrator accounts that may have been created during the attack.
      6. Clean the site — remove malicious code, delete suspicious files, reinstall clean WordPress core.
      7. Update everything — WordPress core, all plugins, all themes. Remove unused plugins and themes.
      8. Harden configuration — apply all recommendations from sections 3 and 4 of this guide.
      9. Rescan — confirm the site is clean before bringing it back online.
      10. Monitor the site — increase monitoring for the next 30 days as attackers often return.
