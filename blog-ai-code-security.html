      Da li je AI-generisan kod bezbedan?

      Konkretna analiza: sta AI alati rade pogresno, realni primeri sa kodom, i kako da proverite pre deploy-a

      1. Kratak odgovor: NE automatski

      AI-generisan kod funkcionise. Kompajlira se, prikazuje lepe stranice, komunicira sa bazom. Ali “radi” i “bezbedan je” su dve potpuno razlicite stvari.

      Evo sta istrazivanja kazu:

      StudijaGodinaKljucni nalaz
      Stanford University2024~40% AI-generisanog koda ima bezbednosne propuste
      NYU & Microsoft2023Programeri sa Copilot-om prihvataju ranjiv kod 30% cesce
      UIUC2024LLM-ovi reprodukuju poznate CVE paterne iz trening podataka
      Google DeepMind2024AI korisnici su previse samouvereni u bezbednost svog koda

      Poslednji nalaz je mozda i najvazniji. Problem nije samo u losem kodu — problem je u tome sto korisnici misle da je kod dobar. “AI zna sta radi” je najopasnije uverenje u vibe coding-u.

      2. 7 konkretnih ranjivosti sa primerima koda

      2.1. Hardkodirani kredencijali

      Najcesci i najopasniji propust. AI rutinski stavlja API kljuceve, lozinke i tokene direktno u izvorni kod.

      AI generise
      // Stripe payment — AI output
      const stripe = require('stripe')(
        'sk_live_51abc...xyz'
      );
      Ispravno
      // Stripe payment — fixed
      const stripe = require('stripe')(
        process.env.STRIPE_SECRET_KEY
      );

      Sta se desava u praksi: GitGuardian je u 2024. detektovao 12.8 miliona novih tajni objavljenih na javnim GitHub repozitorijumima. Botovi skeniraju svaki commit u realnom vremenu. Jedna objavljena Stripe tajna moze biti zloupotrebljena za minutima.

      2.2. SQL Injection

      AI cesto pravi SQL upite spajanjem stringova umesto koristeci parametrizovane upite. Ovo je jedna od najstarijih i najopasnijih ranjivosti — na OWASP Top 10 listi od 2003. godine.

      Ranjivo
      # Python — AI output
      query = f"SELECT * FROM users WHERE email = '{email}'"
      cursor.execute(query)
      Bezbedno
      # Python — parametrizovano
      query = "SELECT * FROM users WHERE email = %s"
      cursor.execute(query, (email,))

      Sa ranjivim kodom, napadac moze da unese u email polje: ' OR '1'='1 i dobije pristup celoj bazi.

      2.3. XSS (Cross-Site Scripting)

      AI cesto koristi nesigurne metode za ubacivanje HTML sadrzaja bez sanitizacije. U React-u, koristi dangerouslySetInnerHTML bez razmisljanja. U vanilla JS-u, koristi innerHTML.

      Ranjivo
      // Prikaz korisnickog komentara
      commentDiv.innerHTML = userComment;
      Bezbedno
      // Prikaz korisnickog komentara
      commentDiv.textContent = userComment;

      Sa innerHTML, napadac moze da ubaci <script> tag koji krade kolacice, sesije, ili preusmerava korisnike na lazne stranice.

      2.4. Nezasticene API rute (IDOR)

      AI pravi API koji radi — ali ne proverava ko pristupa podacima. Klasicna IDOR (Insecure Direct Object Reference) ranjivost.

      Ranjivo
      // GET /api/users/:id — AI output
      app.get('/api/users/:id', (req, res) => {
        const user = db.getUser(req.params.id);
        res.json(user); // svako moze da pristupi
      });
      Bezbedno
      // GET /api/users/:id — sa auth
      app.get('/api/users/:id', authMiddleware, (req, res) => {
        if (req.user.id !== req.params.id)
          return res.status(403).json({error: 'forbidden'});
        const user = db.getUser(req.params.id);
        res.json(user);
      });

      2.5. Slaba CORS konfiguracija

      AI podesava CORS na wildcard (*) jer je to najlaksi nacin da sve radi u development-u. Ali u produkciji, ovo znaci da bilo koji sajt na internetu moze slati zahteve vasem API-ju.

      2.6. Nedostajuca server-side validacija

      AI pravi lepe forme sa frontend validacijom (HTML5 required, pattern, minlength). Ali backend prima podatke bez provere. Napadac zaobilazi frontend za 5 sekundi — otvori DevTools, obrise validaciju, ili koristi curl/Postman.

      # Python — server-side validacija sa Pydantic
      from pydantic import BaseModel, EmailStr, constr

      class UserCreate(BaseModel):
        email: EmailStr
        password: constr(min_length=8, max_length=128)
        name: constr(min_length=1, max_length=100)

      2.7. Detaljne greske u produkciji

      AI cesto ostavlja detaljne error poruke u odgovorima. U development-u ovo pomaze u debug-u. U produkciji, napadac koristi ove informacije da razume vasu infrastrukturu.

      Otkriva detalje
      // Error handler — AI output
      res.status(500).json({
        error: err.message,
        stack: err.stack,
        query: err.sql
      });
      Bezbedno
      // Error handler — fixed
      console.error(err); // loguj interno
      res.status(500).json({
        error: "Internal server error"
      });

      3. Alat po alat: specificni rizici

      Visok rizik

      Bolt / Lovable / v0

      Generisu kompletne projekte. Korisnik nema uvid u kod i ne moze da procene bezbednost. Najrizicniji za produkciju jer se deploy-uju “jednim klikom” bez pregleda.

      Visok rizik

      Replit Agent

      Pravi i deploy-uje automatski. Korisnik cesto ne pregleda generisane fajlove. Posebno opasan za projekte sa placanjem ili bazom podataka.

      Srednji rizik

      GitHub Copilot

      Inline sugestije u editoru. Treniran na javnom GitHub kodu sa losim praksama. Posebno sklon hardkodiranim tajnama i nesigurnim regex paternima.

      Srednji rizik

      Cursor / Claude Code

      Podrzavaju project-level pravila (.cursorrules, CLAUDE.md). Ako definisete bezbednosne standarde, greske su redje. Bez pravila — isti problemi kao ostali alati.

      5. Kako da AI generise bezbedniji kod

      Ne mozete ocekivati bezbednost ako je ne trazite. Evo konkretnih strategija:

      5.1. Bezbednosni prompt template

      Dodajte ovo na kraj svakog prompta kad trazite od AI-ja da napravi novi fajl:

      Bezbednosni zahtevi:
      - Koristi environment varijable za SVE tajne (API kljuceve, lozinke, tokene)
      - Dodaj autentifikaciju/autorizaciju na svaku API rutu
      - Validiraj SVE ulazne podatke na serveru (Zod/Pydantic)
      - CORS: dozvoli samo moj domen, ne wildcard
      - Error handling: loguj interno, vrati genericku poruku korisniku
      - Dodaj rate limiting na javne endpointe

      5.2. Project-level pravila (CLAUDE.md / .cursorrules)

      Ovo su fajlovi koje AI alati citaju pre svakog generisanja. Definisu standarde za ceo projekat:

      # CLAUDE.md ili .cursorrules

      ## Bezbednosni standardi
      - NIKAD ne stavljaj tajne u kod — koristi process.env
      - Svaka API ruta zahteva auth middleware
      - SQL: SAMO parametrizovani upiti, nikad string konkatenacija
      - CORS: allow_origins = ["https://mojsajt.rs"]
      - Svaki korisnicki unos mora biti validiran na serveru
      - Error responses: {error: "Something went wrong"}, NIKAD stack trace
      - Rate limit: 100 req/min po IP na javnim endpointima

      Is AI-Generated Code Secure?

      Concrete analysis: what AI tools get wrong, real code examples, and how to verify before deploy

      1. Short answer: NOT automatically

      AI-generated code works. It compiles, displays beautiful pages, communicates with databases. But “works” and “is secure” are two completely different things.

      Here’s what the research says:

      StudyYearKey finding
      Stanford University2024~40% of AI-generated code has security flaws
      NYU & Microsoft2023Developers with Copilot accept vulnerable code 30% more often
      UIUC2024LLMs reproduce known CVE patterns from training data
      Google DeepMind2024AI users are overconfident about their code’s security

      The last finding might be the most important. The problem isn’t just bad code — the problem is that users think the code is good. “AI knows what it’s doing” is the most dangerous belief in vibe coding.

      2. 7 concrete vulnerabilities with code examples

      2.1. Hardcoded credentials

      The most common and most dangerous flaw. AI routinely places API keys, passwords, and tokens directly in source code.

      AI generates
      // Stripe payment — AI output
      const stripe = require('stripe')(
        'sk_live_51abc...xyz'
      );
      Correct
      // Stripe payment — fixed
      const stripe = require('stripe')(
        process.env.STRIPE_SECRET_KEY
      );

      What happens in practice: GitGuardian detected 12.8 million new secrets published on public GitHub repositories in 2024. Bots scan every commit in real-time. An exposed Stripe secret can be exploited within minutes.

      2.2. SQL Injection

      AI often builds SQL queries by concatenating strings instead of using parameterized queries. This is one of the oldest and most dangerous vulnerabilities — on the OWASP Top 10 list since 2003.

      Vulnerable
      # Python — AI output
      query = f"SELECT * FROM users WHERE email = '{email}'"
      cursor.execute(query)
      Secure
      # Python — parameterized
      query = "SELECT * FROM users WHERE email = %s"
      cursor.execute(query, (email,))

      With vulnerable code, an attacker can enter in the email field: ' OR '1'='1 and gain access to the entire database.

      2.3. XSS (Cross-Site Scripting)

      AI often uses unsafe methods for inserting HTML content without sanitization. In vanilla JS, it uses innerHTML instead of textContent.

      Vulnerable
      // Display user comment
      commentDiv.innerHTML = userComment;
      Secure
      // Display user comment
      commentDiv.textContent = userComment;

      With innerHTML, an attacker can inject <script> tags that steal cookies, sessions, or redirect users to phishing pages.

      2.4. Unprotected API routes (IDOR)

      AI builds APIs that work — but doesn’t check who is accessing the data. Classic IDOR (Insecure Direct Object Reference) vulnerability.

      Vulnerable
      // GET /api/users/:id — AI output
      app.get('/api/users/:id', (req, res) => {
        const user = db.getUser(req.params.id);
        res.json(user); // anyone can access
      });
      Secure
      // GET /api/users/:id — with auth
      app.get('/api/users/:id', authMiddleware, (req, res) => {
        if (req.user.id !== req.params.id)
          return res.status(403).json({error: 'forbidden'});
        const user = db.getUser(req.params.id);
        res.json(user);
      });

      2.5. Weak CORS configuration

      AI sets CORS to wildcard (*) because that’s the easiest way to make everything work in development. In production, this means any website can send requests to your API.

      2.6. Missing server-side validation

      AI builds beautiful forms with frontend validation (HTML5 required, pattern, minlength). But the backend accepts data without any checks. An attacker bypasses the frontend in 5 seconds — open DevTools, remove validation, or use curl/Postman.

      # Python — server-side validation with Pydantic
      from pydantic import BaseModel, EmailStr, constr

      class UserCreate(BaseModel):
        email: EmailStr
        password: constr(min_length=8, max_length=128)
        name: constr(min_length=1, max_length=100)

      2.7. Detailed errors in production

      AI often leaves detailed error messages in responses. In development this helps debugging. In production, attackers use this information to understand your infrastructure.

      Exposes details
      // Error handler — AI output
      res.status(500).json({
        error: err.message,
        stack: err.stack,
        query: err.sql
      });
      Secure
      // Error handler — fixed
      console.error(err); // log internally
      res.status(500).json({
        error: "Internal server error"
      });

      3. Tool by tool: specific risks

      High risk

      Bolt / Lovable / v0

      Generate complete projects. Users have no visibility into the code and can’t assess security. Riskiest for production because they deploy with “one click” without review.

      High risk

      Replit Agent

      Builds and deploys automatically. Users often don’t review generated files. Especially dangerous for projects with payments or databases.

      Medium risk

      GitHub Copilot

      Inline suggestions in the editor. Trained on public GitHub code with bad practices. Particularly prone to hardcoded secrets and insecure regex patterns.

      Medium risk

      Cursor / Claude Code

      Support project-level rules (.cursorrules, CLAUDE.md). If you define security standards, errors are less frequent. Without rules — same problems as other tools.

      5. How to make AI generate more secure code

      You can’t expect security if you don’t ask for it. Here are concrete strategies:

      5.1. Security prompt template

      Add this to the end of every prompt when asking AI to create a new file:

      Security requirements:
      - Use environment variables for ALL secrets (API keys, passwords, tokens)
      - Add authentication/authorization to every API route
      - Validate ALL input on the server (Zod/Pydantic)
      - CORS: allow only my domain, no wildcard
      - Error handling: log internally, return generic message to user
      - Add rate limiting to public endpoints

      5.2. Project-level rules (CLAUDE.md / .cursorrules)

      These are files that AI tools read before every generation. They define standards for the entire project:

      # CLAUDE.md or .cursorrules

      ## Security Standards
      - NEVER put secrets in code — use process.env
      - Every API route requires auth middleware
      - SQL: ONLY parameterized queries, never string concatenation
      - CORS: allow_origins = ["https://mysite.com"]
      - Every user input must be validated on the server
      - Error responses: {error: "Something went wrong"}, NEVER stack trace
      - Rate limit: 100 req/min per IP on public endpoints
