      1. Sta je vibe coding i zasto je eksplodirao

      Andrej Karpathy — bivsi direktor AI-ja u Tesli i jedan od osnivaca OpenAI-ja — skovao je termin “vibe coding” u februaru 2025. u svom tvitu koji je postao viralan. Opisao je kako koristi AI da napravi kompletan projekat bez rucnog pisanja koda: “I just see things, say things, run things, and copy-paste things, and it mostly works.”

      Danas milioni ljudi koriste ovaj pristup. Alati kao sto su Cursor, Claude Code, GitHub Copilot, Replit Agent, Bolt, Lovable i v0 omogucavaju da opissete sta zelite na obicnom jeziku, a AI generise kompletan projekat — frontend, backend, bazu podataka, deploy konfiguraciju.

      Problem? Vecina ovih projekata zavrsi online sa ozbiljnim bezbednosnim propustima. Ne zato sto su AI alati losi, vec zato sto se fokusiraju na jednu stvar: da aplikacija radi. Bezbednost je skoro uvek sekundarna.

      Realan primer: Korisnik je trazio od Cursor-a da napravi SaaS aplikaciju sa Stripe placanjem. AI je generisao funkcionalan sistem za 2 sata. Problem: Stripe secret key je bio hardkodiran u frontend kodu, API rute nisu imale autentifikaciju, i CORS je bio podesen na *. Svako ko otvori DevTools u browseru mogao je da vidi Stripe kljuc i napravi lazne transakcije.

      2. 5 najcescih propusta u vibe coding projektima

      Na osnovu analize hiljada skeniranih sajtova i akademskih istrazivanja (Stanford, NYU, UIUC), ovo je lista propusta sortiranih po ucestalosti i opasnosti:

      2.1. Hardkodirani API kljucevi u kodu

      Ovo je propust #1. Kad kazete AI-ju “povezi se na Supabase” ili “dodaj Stripe placanje”, on cesto stavi kljuceve direktno u JavaScript fajl. Rezultat: svako ko otvori vas sajt i pritisne F12 moze da vidi sve vase tajne.

      Kako AI cesto generise
      // api.js — AI output
      const supabase = createClient(
        'https://xyz.supabase.co',
        'eyJhbGciOiJIUzI1NiIs...service_role_key'
      );
      Kako treba
      // api.js — ispravljeno
      const supabase = createClient(
        process.env.SUPABASE_URL,
        process.env.SUPABASE_SERVICE_KEY
      );

      Zasto je ovo kriticno: Automatski botovi skeniraju GitHub u realnom vremenu. GitGuardian izvestava da se dnevno objavi preko 10,000 novih tajni na javnim repozitorijumima. Cak i ako obrisete tajnu iz koda, ona ostaje u Git istoriji zauvek — osim ako ne koristite git filter-branch ili BFG Repo-Cleaner.

      2.2. Nedostajuci bezbednosni HTTP headeri

      Ovo je najcesci nalaz naseg skenera — pogadja 85% svih skeniranih sajtova. AI alati nikada ne dodaju bezbednosne headere jer se fokusiraju na funkcionalnost, ne na infrastrukturu.

      Headeri koje morate imati:

      HeaderSta radiPrimer vrednosti
      Content-Security-PolicyKontrolise odakle se smeju ucitavati skripte i stilovidefault-src 'self'; script-src 'self'
      Strict-Transport-SecurityForsira HTTPS, brani od downgrade napadamax-age=63072000; includeSubDomains
      X-Frame-OptionsBrani od clickjacking-a (iframe embed)DENY
      X-Content-Type-OptionsBrani od MIME sniffing-anosniff
      Referrer-PolicyKontrolise koje URL info se salje drugim sajtovimastrict-origin-when-cross-origin
      Permissions-PolicyBlokira pristup kameri, mikrofonu, geolokacijicamera=(), microphone=(), geolocation=()

      Konkretna vercel.json konfiguracija za ove headere je opisana u nasem Vercel bezbednosnom vodicu.

      2.3. CORS wildcard — vrata sirom otvorena

      Kada AI pravi API, cesto podesi Access-Control-Allow-Origin: * da bi “sve radilo” u development-u. Problem: u produkciji, ovo znaci da bilo koji sajt na internetu moze da salje zahteve vasem API-ju u ime vasih korisnika.

      Opasno (AI default)
      # FastAPI — AI output
      app.add_middleware(
        CORSMiddleware,
        allow_origins=["*"],
        allow_methods=["*"],
      )
      Bezbedno
      # FastAPI — ispravljeno
      app.add_middleware(
        CORSMiddleware,
        allow_origins=["https://mojsajt.rs"],
        allow_methods=["GET","POST"],
      )

      2.4. Nedostajuca server-side validacija

      AI pravi lepe forme sa frontend validacijom (required, minLength, email format). Ali backend cesto prima podatke bez ikakve provere. Napadac zaobilazi frontend za 5 sekundi — dovoljno je da otvori Postman ili curl.

      Pravilo: Svaka validacija na frontendu je za korisnicko iskustvo. Svaka validacija na backendu je za bezbednost. Obe moraju postojati.

      2.5. Nedostajuca autentifikacija na API rutama

      AI generise API rute koje izgledaju kompletne — vracaju podatke, pisu u bazu — ali nemaju proveru ko salje zahtev. Svako ko zna URL moze da cita i menja podatke drugih korisnika (IDOR ranjivost).

      Test: Otvorite DevTools → Network tab → pogledajte API pozive vaseg sajta. Da li mozete da promenite korisnicki ID u URL-u i dobijete tudje podatke? Ako da, imate IDOR ranjivost.

      3. Kako AI alati prave ove greske

      Da bismo razumeli zasto se ovo desava, treba razumeti kako AI modeli generisu kod:

      • Trenirani su na javnom kodu — GitHub, Stack Overflow, dokumentacija. Taj kod cesto sadrzi lose prakse, zastarele paterne i direktne propuste. AI ih uci i reprodukuje.
      • Optimizovani su za “radi” — AI modeli dobijaju pozitivan signal kad kod kompajlira i radi. Bezbednost se ne testira u tom ciklusu.
      • Kontekst je ogranicen — AI ne vidi vasu infrastrukturu, deployment okruzenje, ili ko ce koristiti API. Generise kod koji radi u izolaciji.
      • Korisnik ne trazi bezbednost — kad kazete “napravi login stranicu”, AI pravi login. Ne dodaje rate limiting, 2FA, CSRF zastitu, ili brute-force detekciju osim ako to eksplicitno ne trazite.

      4. Bezbednosni workflow za vibe coding

      Ovo je workflow koji preporucujemo za svaki vibe coding projekat. Nije komplikovan — 4 koraka pre svakog deploy-a:

      Korak 1: Proverite tajne

      Pre push-a na GitHub, pretrazite ceo projekat za hardkodirane kljuceve:

      # Pretrazite projekat za potencijalne tajne
      grep -r "sk_live\|sk_test\|apikey\|secret\|password\|token" --include="*.js" --include="*.ts" --include="*.py" --include="*.env"

      # Proverite da je .env u .gitignore
      cat .gitignore | grep ".env"

      Ako nadjete bilo sta — premestite u .env fajl i dodajte .env u .gitignore.

      Korak 2: Dodajte bezbednosne headere

      Za Vercel projekte, dodajte u vercel.json:

      {
        "headers": [{
          "source": "/(.*)",
          "headers": [
            { "key": "X-Frame-Options", "value": "DENY" },
            { "key": "X-Content-Type-Options", "value": "nosniff" },
            { "key": "Strict-Transport-Security", "value": "max-age=63072000; includeSubDomains" },
            { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }
          ]
        }]
      }

      Korak 3: Proverite CORS i autentifikaciju

      Otvorite svaku API rutu i proverite: (1) da li ima auth proveru, (2) da li CORS dozvoljava samo vas domen, (3) da li validira ulazne podatke na serveru.

      Korak 4: Pokrenite automatski sken

      Otvorite nas skener, unesite URL i za 30 sekundi dobijate izvestaj sa 240+ provera. Svaki pronadjen problem ima objasnjenje i uputstvo za popravku.

      5. Kako da AI generise bezbedniji kod

      AI alati generisu ono sto trazite. Ako u promptu ne pomenete bezbednost, nema je. Evo kako da promenite to:

      5.1. Bezbednosni kontekst u promptu

      Umesto: “Napravi mi API za korisnike”

      Napiste: “Napravi mi API za korisnike. Koristi environment varijable za sve tajne. Dodaj autentifikaciju na sve rute. Validiraj sve ulazne podatke sa Zod/Pydantic. Podesi CORS samo na moj domen. Dodaj rate limiting.”

      5.2. Project-level pravila

      Ako koristite Cursor, napravite .cursorrules fajl. Za Claude Code, koristite CLAUDE.md. Primer:

      # CLAUDE.md — bezbednosna pravila za projekat

      ## Bezbednosna pravila
      - NIKAD ne hardkoduj API kljuceve, lozinke ili tokene u kod
      - Koristi environment varijable za sve tajne
      - Svaka API ruta mora imati autentifikaciju
      - Validiraj SVE ulazne podatke na serveru
      - CORS: dozvoli samo https://mojsajt.rs
      - Dodaj rate limiting na sve javne endpointe
      - Ne vracaj detaljne greske u produkciji

      5.3. Pregledajte kriticne delove

      Ne morate pregledati svaku liniju. Fokusirajte se na 4 kriticna mesta:

      • Autentifikacija — login, registracija, sesije, tokeni
      • API rute — ko moze da pristupi, sta moze da uradi
      • Baza podataka — SQL upiti, pristupni kljucevi
      • Placanje — Stripe/LemonSqueezy kljucevi, webhook verifikacija

      1. What is vibe coding and why it exploded

      Andrej Karpathy — former head of AI at Tesla and OpenAI co-founder — coined the term “vibe coding” in February 2025 in a viral tweet. He described using AI to build a complete project without writing code manually: “I just see things, say things, run things, and copy-paste things, and it mostly works.”

      Today millions of people use this approach. Tools like Cursor, Claude Code, GitHub Copilot, Replit Agent, Bolt, Lovable, and v0 let you describe what you want in plain language, and AI generates a complete project — frontend, backend, database, deploy configuration.

      The problem? Most of these projects end up online with serious security flaws. Not because AI tools are bad, but because they focus on one thing: making the app work. Security is almost always secondary.

      Real example: A user asked Cursor to build a SaaS app with Stripe payments. AI generated a functional system in 2 hours. The problem: the Stripe secret key was hardcoded in frontend code, API routes had no authentication, and CORS was set to *. Anyone who opened DevTools could see the Stripe key and make fraudulent transactions.

      2. The 5 most common flaws in vibe coding projects

      Based on analysis of thousands of scanned sites and academic research (Stanford, NYU, UIUC), here are the flaws sorted by frequency and severity:

      2.1. Hardcoded API keys in code

      This is flaw #1. When you tell AI “connect to Supabase” or “add Stripe payments,” it often places keys directly in JavaScript files. Result: anyone who opens your site and presses F12 can see all your secrets.

      How AI often generates
      // api.js — AI output
      const supabase = createClient(
        'https://xyz.supabase.co',
        'eyJhbGciOiJIUzI1NiIs...service_role_key'
      );
      How it should be
      // api.js — fixed
      const supabase = createClient(
        process.env.SUPABASE_URL,
        process.env.SUPABASE_SERVICE_KEY
      );

      Why this is critical: Automated bots scan GitHub in real-time. GitGuardian reports that over 10,000 new secrets are published daily on public repositories. Even if you delete a secret from code, it remains in Git history forever — unless you use git filter-branch or BFG Repo-Cleaner.

      2.2. Missing security HTTP headers

      This is our scanner’s most common finding — affecting 85% of all scanned sites. AI tools never add security headers because they focus on functionality, not infrastructure.

      Headers you must have:

      HeaderWhat it doesExample value
      Content-Security-PolicyControls where scripts and styles can load fromdefault-src 'self'; script-src 'self'
      Strict-Transport-SecurityForces HTTPS, protects against downgrade attacksmax-age=63072000; includeSubDomains
      X-Frame-OptionsProtects against clickjacking (iframe embedding)DENY
      X-Content-Type-OptionsPrevents MIME sniffingnosniff
      Referrer-PolicyControls what URL info is sent to other sitesstrict-origin-when-cross-origin
      Permissions-PolicyBlocks access to camera, microphone, geolocationcamera=(), microphone=(), geolocation=()

      The exact vercel.json configuration for these headers is described in our Vercel security guide.

      2.3. CORS wildcard — doors wide open

      When AI builds an API, it often sets Access-Control-Allow-Origin: * so “everything works” in development. The problem: in production, this means any website on the internet can send requests to your API on behalf of your users.

      Dangerous (AI default)
      # FastAPI — AI output
      app.add_middleware(
        CORSMiddleware,
        allow_origins=["*"],
        allow_methods=["*"],
      )
      Secure
      # FastAPI — fixed
      app.add_middleware(
        CORSMiddleware,
        allow_origins=["https://mysite.com"],
        allow_methods=["GET","POST"],
      )

      2.4. Missing server-side validation

      AI builds beautiful forms with frontend validation (required, minLength, email format). But the backend often accepts data without any checks. An attacker bypasses frontend validation in 5 seconds — just open Postman or curl.

      Rule: Every frontend validation is for user experience. Every backend validation is for security. Both must exist.

      2.5. Missing authentication on API routes

      AI generates API routes that look complete — they return data, write to the database — but don’t check who is sending the request. Anyone who knows the URL can read and modify other users’ data (IDOR vulnerability).

      Test: Open DevTools → Network tab → look at your site’s API calls. Can you change the user ID in the URL and get someone else’s data? If yes, you have an IDOR vulnerability.

      3. How AI tools make these mistakes

      To understand why this happens, you need to understand how AI models generate code:

      • Trained on public code — GitHub, Stack Overflow, documentation. That code often contains bad practices, outdated patterns, and direct flaws. AI learns and reproduces them.
      • Optimized for “works” — AI models get positive signals when code compiles and runs. Security isn’t tested in that cycle.
      • Context is limited — AI doesn’t see your infrastructure, deployment environment, or who will use the API. It generates code that works in isolation.
      • User doesn’t ask for security — when you say “build a login page,” AI builds a login. It doesn’t add rate limiting, 2FA, CSRF protection, or brute-force detection unless you explicitly ask.

      4. Security workflow for vibe coding

      This is the workflow we recommend for every vibe coding project. It’s not complicated — 4 steps before every deploy:

      Step 1: Check for secrets

      Before pushing to GitHub, search your entire project for hardcoded keys:

      # Search project for potential secrets
      grep -r "sk_live\|sk_test\|apikey\|secret\|password\|token" --include="*.js" --include="*.ts" --include="*.py" --include="*.env"

      # Verify .env is in .gitignore
      cat .gitignore | grep ".env"

      If you find anything — move it to a .env file and add .env to .gitignore.

      Step 2: Add security headers

      For Vercel projects, add to vercel.json:

      {
        "headers": [{
          "source": "/(.*)",
          "headers": [
            { "key": "X-Frame-Options", "value": "DENY" },
            { "key": "X-Content-Type-Options", "value": "nosniff" },
            { "key": "Strict-Transport-Security", "value": "max-age=63072000; includeSubDomains" },
            { "key": "Referrer-Policy", "value": "strict-origin-when-cross-origin" }
          ]
        }]
      }

      Step 3: Check CORS and authentication

      Open every API route and verify: (1) it has an auth check, (2) CORS only allows your domain, (3) it validates input data on the server.

      5. How to make AI generate more secure code

      AI tools generate what you ask for. If you don’t mention security in your prompt, there won’t be any. Here’s how to change that:

      5.1. Security context in prompts

      Instead of: “Build me a user API”

      Write: “Build me a user API. Use environment variables for all secrets. Add authentication to all routes. Validate all input with Zod/Pydantic. Set CORS to my domain only. Add rate limiting.”

      5.2. Project-level rules

      If you use Cursor, create a .cursorrules file. For Claude Code, use CLAUDE.md. Example:

      # CLAUDE.md — security rules for project

      ## Security Rules
      - NEVER hardcode API keys, passwords, or tokens in code
      - Use environment variables for all secrets
      - Every API route must have authentication
      - Validate ALL input data on the server
      - CORS: allow only https://mysite.com
      - Add rate limiting to all public endpoints
      - Don't return detailed errors in production

      5.3. Review critical parts

      You don’t have to review every line. Focus on 4 critical areas:

      • Authentication — login, registration, sessions, tokens
      • API routes — who can access, what they can do
      • Database — SQL queries, access keys
      • Payments — Stripe/LemonSqueezy keys, webhook verification
